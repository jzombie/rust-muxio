# Plan: mpsc Channel Adapter for muxio Streaming

## Goal

Eliminate the `RpcStreamEvent`/`StreamResponder`/`RpcStreamEncoder` surface area for users who just want "write bytes, read bytes" over a multiplexed RPC stream.

## Current Architecture (what needs wrapping)

### Server side streaming handler
- `endpoint.register_stream_handler(id, |event: RpcStreamEvent, responder: StreamResponder, ctx|)`
- Handler receives `Header`, `PayloadChunk`, `End`, `Error` events
- Uses `responder.respond(bytes, is_finalized)` to reply

### Client side streaming call
- `client.call_rpc_streaming(request, DynamicChannelType)` returns `(RpcStreamEncoder, DynamicReceiver)`
- User manually calls `encoder.write_bytes()`, `encoder.flush()`, `encoder.end_stream()`
- Reads `DynamicReceiver` (Stream of `Result<Vec<u8>, Error>`)

### Key internals
- `RpcRespondableSession` routes `RpcStreamEvent`s to per-request handlers via `stream_method_router`
- `RpcDispatcher::call()` creates an encoder and optionally registers a response handler (`RpcResponseHandler = FnMut(RpcStreamEvent)`)
- `StreamResponder` buffers writes until `read_bytes` installs an encoder-backed writer

## Proposed API

New crate: `extensions/muxio-mpsc-adapter/`

### Server side

```rust
use muxio_mpsc_adapter::ChannelEndpointExt;

let (tx, rx) = mpsc::channel::<Vec<u8>>(64);
endpoint.register_channel_handler(MY_STREAM_ID, tx)?;

// rx is a plain tokio::mpsc::Receiver<Vec<u8>>
// spawn a task to read streamed request chunks:
tokio::spawn(async move {
    while let Some(chunk) = rx.recv().await {
        // process chunk -- no RpcStreamEvent, no StreamResponder
    }
});
```

The adapter:
1. Wraps the handler as `|event, responder, ctx|` that extracts `PayloadChunk` bytes -> `tx.send()`
2. Drops `tx` on `End` or `Error` events so `rx.recv()` returns `None`

### Client side

```rust
use muxio_mpsc_adapter::ChannelCallerExt;

let (writer, mut reader) = client.open_channel(MY_STREAM_ID, 0).await?;
// writer: UnboundedSender<Vec<u8>> -- send request chunks, never blocks
writer.send(b"hello".to_vec()).unwrap();
// reader: UnboundedReceiver<Result<Vec<u8>, RpcServiceError>> -- read response chunks
while let Some(chunk) = reader.recv().await {
    // handle chunk
}
```

The adapter:
1. Internally calls `dispatcher.call()` with `is_finalized: false`
2. Creates a response handler closure that converts `PayloadChunk` -> `tx.send(Ok(bytes))` and `End`/`Error` -> drops tx
3. Spawns a background task that reads from `writer`'s corresponding `rx`, calls `encoder.write_bytes()` + `encoder.flush()`, and `encoder.end_stream()` on channel close

## Implementation details

### `register_channel_handler(method_id, tx: Sender<Vec<u8>>)`

Wraps `register_stream_handler` with a closure that:
- Ignores `Header` events
- Forwards `PayloadChunk.bytes` to `tx.try_send()`
- On `End`/`Error`, drops `tx` so receiver gets `None`
- Ignores `StreamResponder` (channel handler is one-directional: incoming bytes only)

### `open_channel(method_id)` on client

1. Creates `(req_tx, req_rx)` and `(resp_tx, resp_rx)` mpsc channels
2. Calls `call_rpc_streaming` with `is_finalized: false`
3. **Important**: flush the encoder once after `dispatcher.call()` returns — this ensures the initial RPC header/metadata is emitted onto the transport *before* the background task starts. Otherwise the server never sees the call start.
4. Spawns a task: reads from `req_rx` -> writes to encoder + flushes; on close, calls `end_stream`
   - This task naturally handles the encoder lifecycle; no manual flush needed by the user.
5. The response handler (passed as `on_response` to `dispatcher.call`) converts `PayloadChunk` -> `resp_tx.send(Ok(bytes))` and `End`/`Error` -> drops `resp_tx`
6. Returns `(req_tx, resp_rx)`

## Why NOT to put this in core

- `core` is sync/WASM-compatible; mpsc is fundamentally async+tokio
- The adapter is a convenience layer, not protocol logic

## Channel types

Both bounded (`mpsc::channel(n)`) and unbounded (`mpsc::unbounded_channel()`) are supported.

`register_channel_handler` accepts `mpsc::Sender<Vec<u8>>` (bounded). Users who prefer unbounded can wrap with a large buffer or we can add a sibling `register_unbounded_channel_handler`.

On the client side, `open_channel` internally uses bounded channels with a configurable buffer size, matching the existing `DynamicChannelType` semantics.

## Why a new crate instead of adding to existing crates

- Existing crates (`muxio-rpc-service-endpoint`, `muxio-rpc-service-caller`) would gain tokio-only deps
- Keep clean separation: existing API unchanged, adapter is opt-in
