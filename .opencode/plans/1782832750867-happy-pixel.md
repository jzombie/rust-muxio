# Plan: Tests + README for Streaming Handler Registration

## Context

We've implemented `register_stream_handler()` on the endpoint and the
`stream_method_router` on `RpcRespondableSession`. Now we need:

1. **Integration tests** in `extensions/muxio-ext-test/` that exercise
   streaming handlers across all transports (IPC, WS, WASM)
2. **Unit tests** in `extensions/muxio-rpc-service-endpoint/tests/`
3. **README updates** with streaming handler registration examples

## Design decisions (user-confirmed)

- **Test method ID**: `rpc_method_id!("streaming.capture")` — deterministic hash,
  same zero-collision mechanism used by every real service definition
- **Phase 1 (now)**: Event-receipt only — verify Handler/Header/PayloadChunk/End arrive at the
  streaming handler. No response path testing.
- **Phase 2 (future)**: Dispatcher-based response streaming — expose `RpcDispatcher::respond()` to
  the handler so it can send properly framed response chunks that the client's `call_rpc_streaming`
  demux engine can route.
- **Why not raw emit**: `Box<dyn RpcEmit>` = raw transport sink. Calling it directly sends un-framed
  bytes to the socket. The client dispatcher would interpret them as decode errors or noise.
  The correct path requires going through `RpcDispatcher::respond(request_id, payload, is_finalized)`
  which wraps the payload in the proper muxio frame header.

## Existing test architecture (muxio-ext-test)

- `TestTransport` trait in `test_transport.rs` — each transport (ipc, ws,
  wasm) implements `connect()`, `connect_s2c()`, `connect_fail()`,

- `connect()` registers prebuffered Add, Mult, Echo, 0xBAD on server;
  returns `(Arc<Client>, Arc<ClientEndpoint>)`
- `connect_s2c()` registers prebuffered Echo on server (for client-initiated),
  and returns `(Arc<Client>, Arc<ClientEndpoint>, S2cHandle)` where the
  endpoint is the **client's** endpoint (for server-initiated calls)
- Macros in `lib.rs`: `prebuffered_roundtrip_tests!`, `server_to_client_tests!`,
  `transport_state_tests!`
- Test suite functions in `test_suites.rs`
- `build.rs` auto-discovers `tests/*.rs`
- Existing test files: `prebuffered_roundtrip_tests.rs`,
  `prebuffered_server_to_client_tests.rs`, `transport_state_tests.rs`,
  `muxio-tokio-rpc-server/proxy_error_propagation_tests.rs`

## Key constraint

`connect()` returns the **client's** endpoint, not the server's. The server
endpoint (where handlers must be registered) is created and dropped inside
`connect()` after registration. There's no way to register additional
handlers on the server endpoint after `connect()` returns.

## Solution: Add `connect_for_streaming()` to `TestTransport` trait

A new trait method that each transport implements. It creates the server,
registers both prebuffered handlers (Add, Mult, 0xBAD) AND a streaming
handler (for a custom method_id, not Echo::METHOD_ID to avoid conflicts),
then returns client + client endpoint.

The streaming handler uses `rpc_method_id!("streaming.capture")` — a
deterministic hash that can't collide with any `rpc_method_id!("domain.method")`.

### Files to modify

#### 1. `endpoint_helpers.rs` — Add `register_stream_capture_handler()`

```rust
use muxio_rpc_service::rpc_method_id;

/// Stable, collision-free method ID for streaming capture tests.
pub const STREAMING_CAPTURE_METHOD_ID: u64 = rpc_method_id!("streaming.capture");

/// Register a streaming handler at `STREAMING_CAPTURE_METHOD_ID` that captures
/// received events into a shared Vec for test verification.
/// Returns an Arc<Mutex<Vec<RpcStreamEvent>>> that the test can inspect.
pub async fn register_stream_capture_handler<C: Send + Sync + Clone + 'static>(
    endpoint: &impl RpcServiceEndpointInterface<C>,
) -> Arc<Mutex<Vec<RpcStreamEvent>>> {
    let events = Arc::new(Mutex::new(Vec::new()));
    let captured = events.clone();
    endpoint.register_stream_handler(STREAMING_CAPTURE_METHOD_ID, move |event, _emit, _ctx| {
        captured.lock().unwrap().push(event);
    }).await.unwrap();
    events
}
```

The handler captures every received event into a `Vec<RpcStreamEvent>`. The
test can later assert what was received: one Header, N PayloadChunks, one End.

## Phase 2 (future): Dispatcher-based response streaming

After Phase 1 event-receipt tests pass, the response path needs the
streaming handler to be able to call `RpcDispatcher::respond()` for
properly framed responses. Design:

1. **Pass dispatcher reference** to the streaming handler closure
   — either add a `&mut RpcDispatcher` parameter or expose a respond
   method on a new wrapper type
2. **`respond(request_id, chunk, is_finalized)`** wraps the payload in a
   proper muxio response frame (matching request_id, correct frame type)
3. **Client receives PayloadChunk/End** events demuxed by request_id,
   populating the `DynamicReceiver` from `call_rpc_streaming()`
4. **Test**: register streaming echo handler, send chunks, verify chunks
   echoed back via receiver

```markdown
### Handling streaming requests on the server

Streaming handlers are registered via `register_stream_handler()` and receive
individual `RpcStreamEvent`s as they arrive from the transport. Unlike
prebuffered handlers (which accumulate the entire request then invoke an
async callback), streaming handlers are called synchronously for each event:

```rust
use muxio_core::rpc::rpc_internals::RpcStreamEvent;
use muxio_ext_test::endpoint_helpers::STREAMING_CAPTURE_METHOD_ID;
use muxio_rpc_service_endpoint::RpcServiceEndpointInterface;

endpoint.register_stream_handler(STREAMING_CAPTURE_METHOD_ID, |event, _emit, _ctx| {
    match event {
        RpcStreamEvent::Header { rpc_method_id, .. } => {
            println!("Stream started for method {rpc_method_id}");
        }
        RpcStreamEvent::PayloadChunk { bytes, .. } => {
            println!("Received {} bytes", bytes.len());
        }
        RpcStreamEvent::End { .. } => {
            println!("Stream complete");
        }
        RpcStreamEvent::Error { frame_decode_error, .. } => {
            eprintln!("Stream error: {frame_decode_error:?}");
        }
    }
}).await.unwrap();
```

> **Note:** The second argument (`_emit`) accepts a raw transport byte sink
> (`Box<dyn RpcEmit>`). Sending un-framed bytes through it will not be
> properly routed by the client's demux engine. A future API will expose
> `RpcDispatcher::respond()` to stream properly framed response chunks.
```

No response-path example until Phase 2 is implemented.
