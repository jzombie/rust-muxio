# Plan: Add Streaming Handler Registration to Muxio Endpoint

## Answer to user's question

**Yes, OpenCode is correct.** Muxio 0.11.0-alpha has NO streaming handler registration on the endpoint/server side. Only `register_prebuffered()` exists. The core transport layer supports streaming (`RpcDispatcher`, `RpcSession`, `RpcStreamEncoder`, `RpcStreamEvent`), but the `RpcServiceEndpointInterface` abstraction only exposes buffered handlers.

## Current term-session-server workaround

The `term-session-server` handles PTY output by polling every 8ms and pushing via `ctx.call_rpc_buffered(PushOutput::METHOD_ID, ...)` with custom binary framing. Adding streaming handler support to Muxio would allow true bidirectional streaming, removing the polling overhead and custom framing.

## Design decisions (user-confirmed)

1. **API**: `register_stream_handler(method_id, Fn(RpcStreamEvent, EmitFn, C) -> Future<Result<()>>)` — event-driven callback style
2. **Storage**: Separate `HashMap<u64, RpcStreamHandler<C>>` alongside existing prebuffered map
3. **Version**: `0.11.1-alpha` (patch bump)

## Handler signature detail

```rust
// RpcStreamEvent variants:
//   Header { rpc_request_id, rpc_header, stream_encoder }
//   PayloadChunk { rpc_request_id, bytes }
//   End { rpc_request_id }
//   Cancel { rpc_request_id }
//   Error { rpc_header, rpc_request_id, rpc_method_id, frame_decode_error }

// Streaming handler:

async fn my_stream_handler(
    event: RpcStreamEvent,
    emit: Box<dyn RpcEmit + Send + Sync>,
    ctx: Arc<RpcIpcConnectionContext>,
) -> Result<(), Box<dyn Error + Send + Sync>> {
    match event {
        RpcStreamEvent::Header { rpc_header, .. } => {
            // Initialize per-stream state
        }
        RpcStreamEvent::PayloadChunk { bytes, .. } => {
            // Process streaming chunk
            // Optionally send response chunk via emit(&chunk)
        }
        RpcStreamEvent::End { .. } => {
            // Stream complete
        }
        _ => {}
    }
    Ok(())
}
```

The `EmitFn` wraps `RpcDispatcher::respond()` — it's how the server sends response frames back to the client mid-stream (for streaming responses).

Alternatively, we could pass an `RpcStreamEncoder` that the handler calls `write_bytes()` on.

## Dispatch changes in `read_bytes()`

Current flow:
```
read_bytes() → dispatcher.read_bytes() → collect finalized requests → execute prebuffered handlers → dispatcher.respond()
```

New flow:
```
read_bytes() → dispatcher.read_bytes()
  → for each event decoded:
    → check if method_id has streaming handler
    → YES: forward event + emit_fn to streaming handler
    → NO (prebuffered): accumulate buffer as before
  → for finalized prebuffered requests: execute handler → respond()
```

This means `read_bytes()` needs to process not just finalized requests, but also streaming events as they arrive. The dispatcher already processes events internally via `init_catch_all_response_handler`; the question is how to intercept them before they get buffered.

Actually, looking more closely at the dispatcher internals — `RpcDispatcher::read_bytes()` calls `self.rpc_respondable_session.read_bytes()` which triggers the catch-all response handler that accumulates everything. To support streaming on the endpoint, we need to either:
- **Option A**: Use the existing per-request `on_response` handler instead of the catch-all. When we detect a streaming method, install a custom handler that forwards events to the user's stream handler.
- **Option B**: Add a pre-processing hook to `RpcRespondableSession` that checks method_id before the catch-all accumulates.
- **Option C**: Change the endpoint's `read_bytes()` to first peek at method_id, and for streaming methods, register a per-request `RpcResponseHandler` that forwards events directly instead of buffering.

**Option C is simplest.** The `RpcDispatcher::call()` and `respond()` methods already support per-request `on_response` handlers. For incoming requests, the endpoint currently relies on the catch-all buffer. For streaming, we can intercept earlier using the dispatcher's `init_catch_all_response_handler`-like mechanism but with a routing decision.

Actually, the cleanest approach: modify `RpcRespondableSession` to accept a **stream routing callback** — a closure `Fn(u64) -> Option<StreamRoute>` where `StreamRoute` is either `Prebuffered` or `Stream`. Based on this, the session's catch-all handler either accumulates (prebuffered) or forwards events directly to the registered streaming handler.

Even simpler: keep the catch-all accumulation but add a separate event listener that fires BEFORE the catch-all. For streaming methods, the listener forwards events AND marks them as "don't accumulate". But this requires invasive changes to `RpcRespondableSession`.

Let me think of the minimal change approach...

### Minimal change approach:

1. Add `HashMap<u64, RpcStreamHandler<C>>` to endpoint
2. Add `init_stream_dispatch_handler()` to `RpcDispatcher` or `RpcRespondableSession` that installs a per-stream handler for known streaming methods
3. When `read_bytes()` processes events, for streaming methods, forward events directly and skip the buffer accumulation
4. The `RpcRespondableSession` needs a mechanism to route events either to the catch-all buffer OR to a per-stream handler

Actually, looking at `RpcRespondableSession`:

```rust
pub fn set_catch_all_response_handler(&mut self, handler: Box<RpcResponseHandler>)
```

This sets a single catch-all for all streams. If we want to route by method_id, we change the catch-all to a routing handler:

```rust
set_response_routing_handler(
    &mut self,
    router: Box<dyn Fn(u64) -> Option<Box<dyn RpcResponseHandler>>>,
    catch_all: Box<RpcResponseHandler>,
)
```

But that's invasive. Let me check if there's a simpler way...

Actually, re-reading `RpcRespondableSession`:

The `set_catch_all_response_handler` sets a handler that receives `RpcStreamEvent`s for ALL incoming response streams. This is used to accumulate prebuffered requests. But for a **server endpoint**, incoming requests arrive as `RpcMessageType::Call`, not `Response`. Let me re-read the dispatcher...

Actually wait - on the server side, incoming `Call` messages arrive. The dispatcher's catch-all handler processes these as responses from the remote peer. On the SERVER side, when a client makes a call:
- Client sends `Call` frame
- Server receives `Call` frame  
- The session's handler gets `Header`, `PayloadChunk`, `End` events for the incoming call
- The endpoint's catch-all accumulates these into `RpcRequest`s in the queue
- Finalized requests get processed by prebuffered handlers

For streaming, we need the catch-all to NOT accumulate for streaming methods, but instead forward events to the user's streaming handler.

The simplest change: **Replace the catch-all with a routing handler** that checks if the method_id has a streaming handler, and either accumulates (prebuffered) or forwards (streaming).

This means modifying `RpcRespondableSession` (or `RpcDispatcher`) to support a routing callback.
